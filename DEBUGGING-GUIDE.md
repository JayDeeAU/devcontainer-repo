# Debugging Guide — General Concepts

**For projects using ucm and multi-environment setup**

> **Note:** This is a general-purpose debugging reference shared across projects via the `.devcontainer` submodule.
> For project-specific ports, host IPs, and stack details, see your project's rendered `docs/DEBUGGING.md`
> (generated by `infra-init.sh` from `.project-config.json`).

---

## Quick Start

### Start Development Environment
```bash
# Start your local development environment
env-local

# Check status
env-status

# View logs
env-logs [service-name]
```

### Attach Debugger
1. Ensure environment is running: `env-status`
2. Open VS Code Run & Debug panel (Ctrl+Shift+D)
3. Select appropriate debug configuration
4. Press F5 or click green play button

## Environment Overview

This project uses three isolated environments with non-overlapping port ranges defined in `.project-config.json`:

| Environment | Branch Pattern | Purpose |
|-------------|----------------|---------|
| **Local** | `feature/*`, `hotfix/*` | Active development |
| **Staging** | `develop` | Pre-production testing |
| **Production** | `main` | Deployment monitoring |

### Common Commands

```bash
# Environment Management
env-local              # Start local development
env-staging            # Start staging environment
env-prod               # Start production environment

# Debug Branch Workflow
git switch main        # Switch to branch you want to debug
dbs issue-name         # Create debug branch from current branch
dbl                    # List active debug branches
dbf                    # End debug session, return to parent branch

# Utilities
env-status             # Show all environments
env-health             # Health check current environment
env-stop [env]         # Stop specific or all environments
env-logs [service]     # View service logs
```

## VS Code Debugging

### Launch Configurations

Your `.vscode/launch.json` is **rendered by `infra-init.sh`** with the correct ports from `.project-config.json`. Do not hardcode ports — re-render if ports change.

Typical configurations include:

1. **Backend Debugging** (Python/debugpy)
   ```jsonc
   {
     "name": "Local: Backend FastAPI",
     "type": "debugpy",
     "request": "attach",
     "connect": {
       "host": "<host-ip>",       // from network.host_ip
       "port": "<debug-port>"     // from ports.local.backend_debug
     },
     "pathMappings": [{
       "localRoot": "${workspaceFolder}/backend",
       "remoteRoot": "/app"
     }]
   }
   ```

2. **Frontend Debugging** (Node.js inspect)
   ```jsonc
   {
     "name": "Local: Next.js Frontend",
     "type": "node",
     "request": "attach",
     "port": "<debug-port>",       // from ports.local.frontend_debug_node
     "address": "<host-ip>",       // from network.host_ip
     "restart": true,
     "localRoot": "${workspaceFolder}/frontend",
     "remoteRoot": "/app"
   }
   ```

3. **Compound Configurations** (Full Stack)
   ```jsonc
   {
     "name": "Full Stack Debug",
     "configurations": [
       "Local: Backend FastAPI",
       "Local: Next.js Frontend"
     ],
     "stopAll": true
   }
   ```

## Debug Branch Workflow

### Concept
Debug branches are **temporary scratch branches** for investigating production/staging issues with full source mounting in local environment.

### When to Use
- Investigating production bugs
- Testing staging issues locally
- Reproducing specific version behavior
- Testing hotfix before creating official hotfix branch

### Usage Workflow
```bash
# 1. Switch to the branch you want to debug
git switch main           # For production issues
git switch develop        # For staging issues
git switch feature/auth   # For feature branch issues

# 2. Create debug branch from current branch
dbs payment-timeout
# Creates: debug/main-payment-timeout-20251021-143022

# 3. Environment auto-starts with source mounting
# Make changes, add debug prints, test fixes

# 4. List active debug branches
dbl

# 5. When done, end debug session
dbf
# Returns to parent branch (main/develop/feature/*)
# Deletes debug branch
```

### Debug Branch Rules
- Temporary branches for testing only
- Auto-deleted by dbf command
- Full source mounting in local environment
- Can commit changes if needed (optional)
- Not for long-term development — use feature branches

## Technology-Specific Debugging

### Next.js / React (Frontend)

**Dev scripts** (in `frontend/package.json`):

| Script | Purpose |
|--------|---------|
| `pnpm dev` | Development with Node.js inspector enabled |
| `pnpm dev:no-debug` | Development without inspector (faster startup) |
| `pnpm dev:wait` | Development with inspector in break-on-start mode (`--inspect-brk`) |

The container's internal inspector port is always `9229`. The host-mapped port is defined in `.project-config.json` as `ports.local.frontend_debug_node` and may differ per project to avoid conflicts.

**Breakpoint tips:**
- Server Components: breakpoints work in `.tsx` files
- API Routes: breakpoints work in `route.ts` files
- Client Components: use Chrome DevTools (not VS Code attach)

### Python / FastAPI (Backend)

The backend container exposes debugpy on internal port `5678`, mapped to the host port defined in `ports.local.backend_debug`.

**VS Code** auto-attaches via the rendered launch.json config.

Set `justMyCode: false` in launch.json to step into library code.

### Node.js / Express (Backend)

**Enable inspector:**
```json
{
  "scripts": {
    "dev": "NODE_OPTIONS='--inspect=0.0.0.0:9229' nodemon server.js"
  }
}
```

## Troubleshooting

### "Cannot connect to debug port"

1. Check service is running: `env-status`
2. Check debug port is exposed: `docker ps --format 'table {{.Names}}\t{{.Ports}}'`
3. Check logs: `env-logs <service>`
4. Verify host IP matches `network.host_ip` in `.project-config.json`

### "Breakpoints not working" (hollow circles)

1. Verify source maps are enabled (default in dev mode)
2. Check `localRoot`/`remoteRoot` path mappings in launch.json
3. Ensure `justMyCode: false` for library debugging (Python)
4. Restart debugger: Ctrl+Shift+F5

### "env-local fails on host"

1. Create required Docker networks:
   ```bash
   docker network create <project>-network-local
   docker network create redis_admin_network  # if used
   ```
2. Check `.project-config.json` for network requirements

### "Source maps not resolving"

1. **Frontend:** Dev mode generates source maps by default
2. **Backend:** Check compiler settings (TypeScript/Babel)
3. Clear build cache: `rm -rf .next` (frontend) or `__pycache__` (Python)

## Best Practices

### Development Workflow
1. Always work in **feature branches**, not main/develop
2. Use `env-local` for active development
3. Test in `env-staging` before merging
4. Use debug branches only for investigating issues

### Debugging Workflow
1. Start environment before attaching debugger
2. Use compound configurations for full-stack debugging
3. Check health endpoints before debugging
4. Add breakpoints in VS Code, not `debugger;` statements

### Performance Tips

**Reduce debug overhead:**
- Use `dev:no-debug` script for performance testing
- Set `justMyCode: true` to skip library code
- Use conditional breakpoints instead of many breakpoints

**Speed up containers:**
- Use Docker layer caching effectively
- Pre-build images: `env-staging --build`
- Use volumes for node_modules and build artifacts
